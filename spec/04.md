# 04. Provider Architecture

## TLDR

Each provider has a dedicated module, explicit capabilities, and a small adapter surface that translates provider-neutral plans into API operations.

## Module Layout

Provider code lives under `src/donazopy/providers/`. Each provider uses a lowercase module name such as `cloudflare.py`, `namecheap.py`, or `google_cloud.py`. Shared contracts live in `base.py` and the registry lives in `registry.py`.

## Adapter Contracts

Adapters should be split by capability: DNS hosting, zone-file import/export, registrar/domain management, and delegation. A provider can implement any subset. Unsupported operations must fail clearly.

## Provider Set

The named providers are Cloudflare, Vercel, AWS, Azure, Google Cloud, IONOS, Joker, GoDaddy, Namecheap, Hosting.com, Hostinger, and Bluehost. Additional relevant providers include DNSimple, DigitalOcean, Hetzner, Linode, Vultr, Gandi, Porkbun, and Dynadot.

## Implementation Rule

Provider API behavior must be confirmed against official documentation before enabling live writes. Unknown or account-specific APIs may exist in the registry as researched placeholders but should not claim tested write support until covered by tests.

## Acceptance Criteria

- A provider registry lists all supported and planned providers.
- Provider capabilities are machine-readable.
- Provider code is isolated by module.
