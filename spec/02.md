# 02. Domain Model

## TLDR

The domain model centers on zones, records, providers, capabilities, credentials, sync plans, and verification results.

## Entities

A zone represents an authoritative DNS namespace with an origin, TTL defaults, SOA record, NS records, and typed resource records. A provider represents an external service with DNS-hosting capabilities, registrar capabilities, or both. A credential profile describes how authentication is loaded without storing secrets in tracked files.

## Record Model

Records must preserve owner name, record type, TTL, class, normalized value, provider identifiers when present, and source location where useful. Provider adapters may need extra metadata, but core operations should use a provider-neutral record representation.

## Capability Model

Capabilities include zone read, zone write, zone export, zone import, delegation read, delegation write, and domain read. Every provider module must declare support explicitly. Commands must fail early if a requested operation requires unsupported capabilities.

## Plan Model

A sync plan contains creates, updates, deletes, unchanged records, warnings, unsafe changes, and verification steps. Plans are the unit of review before writes.

## Acceptance Criteria

- Provider capability checks are available through the CLI.
- Internal APIs distinguish DNS hosting from registrar delegation.
- Sync plans can be produced without contacting write endpoints.
