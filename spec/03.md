# 03. Zone File Engine

## TLDR

Donazopy uses dnspython as the authoritative parser and serializer for local BIND-style zone files.

## Parsing

Zone parsing must support file and text inputs. It should require an origin when it cannot infer one safely, validate SOA and NS expectations, reject empty input, and surface parse failures as Donazopy-specific errors with readable messages.

## Normalization

The engine should normalize names, record values, TTLs, and classes while preserving enough information for stable serialization. It should support common DNS record types including SOA, NS, A, AAAA, CNAME, MX, TXT, SRV, CAA, PTR, and DNSSEC records where dnspython supports them.

## Serialization

Serialization should produce stable output for review and version control. SOA serial handling must be explicit: preserve by default, optionally bump for changed zones, and never silently invent production serials.

## Diffing

The zone engine should compare normalized local zone state with provider state and produce a plan. CNAME conflicts, apex constraints, duplicate records, and provider-specific unsupported values must become warnings or blocking errors before writes.

## Acceptance Criteria

- Valid zones parse successfully.
- Invalid zones produce typed errors.
- Serialization is stable enough for deterministic tests.
- Diffs are independent of provider API response ordering.
