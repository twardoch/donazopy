# 10. Safety, Validation, and Observability

## TLDR

Donazopy favors explicit plans, local validation, redacted logs, and post-change verification over silent automation.

## Safety Gates

The tool must validate zone files before provider writes, require dry-run plans for destructive operations, block unsupported provider capabilities, and ask for explicit confirmation before applying unsafe changes.

## Observability

CLI output should identify the domain, provider, operation, number of records affected, warnings, verification steps, and resulting status. Debug logs must be useful without exposing secrets.

## Failure Handling

Provider errors should be categorized as authentication, authorization, missing zone, validation, rate limit, transient network, unsafe change, or unknown provider failures. Transient retry policies can be added after provider-specific behavior is understood.

## Acceptance Criteria

- Secrets are redacted in logs and exceptions.
- Unsafe changes are visible in plans.
- Verification failures do not masquerade as successful applies.
