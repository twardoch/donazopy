# Technical Specification: `donazopy doctor`

**Version:** 1.0  
**Status:** Draft  
**Target CLI:** `donazopy doctor TARGET [OPTIONS]`

---

## 1. Overview

The `doctor` command performs **automated DNS health checks** against a live provider-hosted zone (or, in the future, a local zone file). It detects common misconfigurations, stale records, missing email security records, and other anomalies, then reports them with clear severity levels and actionable advice.

When invoked with `--fix`, the tool attempts to repair issues that are safe to automate (e.g., removal of duplicate records, deletion of stale NS records). For issues that cannot be safely fixed, the command prints precise, copy-paste-ready instructions the user can follow.

## 2. Research & Heuristics Catalogue

A thorough review of DNS operational best practices, RFCs, and real-world migration pain points yields the following diagnostic categories.

### 2.1 Duplicate Records

**Problem:** Records that are semantically identical but differ in presentation (quotation, escaping, whitespace, trailing dots, case). These cause unpredictable behaviour because the auth server returns all matching records; resolvers may merge or rotate them inconsistently.

**Sub-heuristics:**
- **TXT quotation variants:** `"v=spf1 ..."` vs `"\"v=spf1 ...\""` (escaped inner quotes, leading/trailing whitespace after stripping quotes).
- **CNAME trailing dot:** `host.example.com.` vs `host.example.com`.
- **A/AAAA whitespace/padding:** `192.0.2.1 ` vs `192.0.2.1`.
- **NS records with/without trailing dot.**
- **Case sensitivity** where record type is case-insensitive but data isn’t (`MX` vs `mx` as type – ignored; content `mail` vs `MAIL` – for domain names, case-insensitive; for TXT, case-sensitive).

**Normalisation function** `canonical_content(record_type, content)`:
- Strip leading/trailing whitespace.
- For TXT: unquote outer `"`, then unescape inner `\"` → `"`. Re‑canonicalise by joining all character-data.
- For names (CNAME, MX, NS, PTR, SRV target): lower‑case, ensure trailing dot (if not fully qualified, make absolute using zone origin).
- For A/AAAA: trim, canonise IPv6 representation (`::`).
- Return a tuple `(type, name_lower, canonical_content)` for deduplication.

### 2.2 Stale / Incorrect NS Records at Apex

**Problem:** After migrating a zone from another provider (e.g., Ionos → Cloudflare), the zone’s apex still contains NS records pointing to the old provider’s nameservers. Most cloud providers automatically maintain apex NS records; user‑added NS records can create conflicts or confusion.

**Detection:**
1. Fetch authoritative NS set from the **registrar** (if credentials available). This is the delegation.
2. Fetch NS records from the **zone apex** via the DNS provider.
3. Compare:
   - If the zone apex contains NS records that differ from the delegated set, flag as `ERROR`.
   - If registrar data is unavailable, compare zone NS against the provider’s known default nameservers (e.g., `*.ns.cloudflare.com` for Cloudflare). Flag mismatches as `WARNING` (cannot be certain).
4. Also flag if the zone apex **lacks** NS records entirely (unusual for some providers, but possible after import) – `WARNING`.

**Fix:**
- If the provider allows deleting non‑default apex NS records, delete all foreign NS records and let the provider auto‑populate its own.
- If deletion is not supported (API restriction), print a message: *“Remove the following NS records via the provider dashboard: … and ensure only the provider’s assigned nameservers remain.”*

### 2.3 Email Security Hygiene

#### 2.3.1 MX Record Presence
- At least one MX record must exist at the **zone apex** if the domain handles email.
- `ERROR` if no MX and no explicit “no email” indicator (see NULL MX).
- NULL MX: `MX 0 .` (RFC 7505) – perfectly valid; skip further email checks.
- Multiple MX with same priority: `WARNING` (not harmful but untidy).

#### 2.3.2 SPF
- RFC 7208: a domain SHOULD publish an SPF record.  
- Check for a TXT record at apex starting with `v=spf1`.
- If exactly one exists, optionally validate syntax (basic: balanced mechanisms, no unknown).  
- If **zero** → `WARNING` (unless NULL MX).  
- If **multiple** → `ERROR` (violates RFC).  
- **Fix:** Cannot auto‑generate safe SPF. Print a template: `v=spf1 mx -all` and explain how to customise.

#### 2.3.3 DKIM
- Look for TXT records matching `*._domainkey` that start with `v=DKIM1`.
- At least one such record is recommended for outbound email domains.
- If none → `INFO` (or `WARNING` for business‑critical domains, heuristically determined by presence of MX).  
- If a record contains `p=` with a public key, check key length (>=1024 bits); warn if `512`.  
- **Fix:** Cannot generate keys. Output: *“Use your email provider to generate a DKIM key and add a TXT record at selector._domainkey with the public key.”*

#### 2.3.4 DMARC
- RFC 7489: a domain SHOULD publish a DMARC record.
- TXT at `_dmarc` starting with `v=DMARC1`.
- If missing → `WARNING`; if multiple → `ERROR`.  
- If present, parse `p=` tag: `none` → `INFO` (monitoring mode), `quarantine`/`reject` → OK.  
- Check `rua`/`ruf` for valid mailto URIs.  
- **Fix:** For a missing record, we could add a safe `v=DMARC1; p=none; rua=mailto:dmarc@domain` with a warning that it should be monitored and tightened. However, this changes email authentication. Better to print a recommended minimal record and let the user decide. Under `--fix`, we could offer an interactive prompt (out of scope for now). Document instructions.

#### 2.3.5 MTA‑STS (optional)
- TXT record at `_mta-sts` with `v=STSv1`.  
- If MX is present but this is missing → `INFO` (improves TLS security).  
- **Fix:** Instruction: *“Add TXT record `_mta-sts` with `v=STSv1; id=…` and a policy file at `https://mta-sts.<domain>/.well-known/mta-sts.txt`.”*

#### 2.3.6 BIMI (optional)
- TXT at `default._bimi` – `INFO` if missing, when domain has DMARC at `quarantine`/`reject`.  
- **Fix:** instructions only.

### 2.4 CNAME at Zone Apex

**Problem:** RFC 1912 forbids a CNAME at the zone apex coexisting with SOA and NS. Providers typically reject it, but some legacy imports may leave one.

- **Detection:** If apex name has a CNAME record → `ERROR`.  
- **Fix:** Impossible automatically. Instruct: *“Remove the CNAME record at the apex; use A/AAAA records or an ALIAS/ANAME virtual record if your provider supports it.”*

### 2.5 Non‑Recommended TTLs

- TTL < 60 seconds (except for TXT/DNS‑01 challenge records) → `WARNING`.  
- TTL > 86400 seconds for non‑SOA records → `WARNING` (slow propagation changes).  
- **Fix:** `--fix` can reset TTL to a sensible default (e.g., 3600) for records that are not SOA or NS. But changing TTL may be disruptive; perhaps only if explicitly asked? Safer: print suggestions.

### 2.6 Glue Record Consistency (Advanced)

- If apex NS records point to a hostname under the same domain (e.g., `ns1.example.com`), those hostnames need A/AAAA glue records at the registrar (parent zone).  
- Because `donazopy` can talk to registrars, we could cross‑check: for each in‑bailiwick NS name, ensure an A/AAAA record exists in the zone *and* that the registrar has matching glue.  
- Complexity high; initial version: `WARNING` if in‑bailiwick NS lacks corresponding A/AAAA in the zone.  
- **Fix:** Print instructions to add glue via registrar.

### 2.7 Delegation Mismatch

- Compare registrar’s nameserver set with the NS records served by the authoritative provider.  
- If they differ → `ERROR`. This can cause broken resolution.  
- **Fix:** If the registrar provider is available and writable, `--fix` could push the correct NS from the zone to the registrar (or vice versa). By default, trust the provider’s NS and update the registrar. Provide confirmation prompt. If credentials missing, instruct user.

### 2.8 SOA & DNSSEC Sanity (Provider‑Specific)

- Some providers expose SOA parameters; unusual values (refresh < 3600, retry too low) may be flagged as `INFO`.  
- DNSSEC status: if provider supports DNSSEC, check if it’s enabled; if not, `INFO` recommending activation. No auto‑fix.

### 2.9 Wildcard Record Risk

- A wildcard record `*.example.com` exists. If combined with specific records, it’s usually fine, but if it’s a CNAME or points to an external service, it can cause unexpected traffic routing.  
- Heuristic: if `*` is a CNAME → `WARNING`. If `*` exists and no other more‑specific records for common names (www, mail, autodiscover) → `INFO`.  
- **Fix:** none; just report.

### 2.10 Orphan Records (External Reachability)

- Optionally check that A/AAAA records for MX targets, CNAME targets, SRV targets resolve. Requires external resolution, which may be slow and unreliable. Defer to future “deep scan” mode.

## 3. Command Interface

```
donazopy doctor TARGET [OPTIONS]

TARGET: provider/domain (e.g., cloudflare/example.com, godaddy/example.com).
```

**Options:**

| Flag | Description |
|------|-------------|
| `--fix` | Attempt to repair fixable problems. |
| `--dry-run` | With `--fix`, show what would be changed without applying. Implies `--fix`. |
| `--format` | `text` (default), `json`. Machine‑readable output. |
| `--dotenv-path` | Path to `.env` file for credentials. |
| `--yes` / `-y` | Skip confirmation prompts (only with `--fix`). |
| `--checks` | Comma‑separated list of checks to run (e.g., `duplicates,email,ns`). Default: all. |
| `--severity` | Minimum severity to report (`error`, `warning`, `info`). Default: `warning`. |

**Examples:**
```bash
donazopy doctor cloudflare/example.com
donazopy doctor godaddy/example.com --fix --dry-run
donazopy doctor cloudflare/example.com --format json --checks duplicates,ns
```

## 4. Architecture & Implementation Plan

### 4.1 New Package: `donazopy.doctor`

Structure:
```
donazopy/
  doctor/
    __init__.py
    engine.py       # Doctor class, orchestration
    findings.py     # Finding dataclass, severity enum
    checks/
      __init__.py
      base.py       # Abstract Check class
      duplicates.py
      ns_stale.py
      email.py      # MX, SPF, DKIM, DMARC, MTA-STS, BIMI
      cname_apex.py
      ttl.py
      delegation.py
      wildcard.py
      soa.py
      glue.py
    normalizer.py   # canonical_content(), dedup logic
```

### 4.2 Core Classes

#### `Severity` enum: `ERROR`, `WARNING`, `INFO`

#### `Finding` dataclass:
```python
@dataclass
class Finding:
    check_name: str
    severity: Severity
    message: str
    detail: str = ""
    fixable: bool = False
    fix_hint: Optional[str] = None       # human-readable fix instruction
    affected_records: List[dict] = field(default_factory=list)  # provider-agnostic record dicts
    fix_action: Optional[Callable] = None  # function to execute fix (only set if fixable)
```

#### `Check` (abstract base):
```python
class Check(ABC):
    name: str
    severity: Severity = Severity.WARNING

    @abstractmethod
    async def run(self, zone: ZoneSnapshot) -> List[Finding]:
        ...
```

`ZoneSnapshot` contains:
- `domain_name`: str
- `records`: list of parsed record objects (name, type, content, ttl, proxiable, etc.)
- `provider`: DNS provider instance
- `registrar`: optional registrar provider instance
- `provider_spec`: ProviderSpec metadata

### 4.3 Execution Flow

```
Doctor.run(target, fix=False, dry_run=False, ...)
  1. Parse target, obtain DNS provider (and optionally registrar provider).
  2. Fetch all DNS records for the zone → ZoneSnapshot.
  3. Instantiate all enabled Check subclasses.
  4. For each check, run `check.run(snapshot)`, collect Findings.
  5. Filter by severity.
  6. Output:
     - text: formatted table with icons, colour (if terminal), fix hints.
     - json: machine-readable array.
  7. If --fix:
     a. Filter fixable findings.
     b. If --dry-run, show actions and exit.
     c. Ask for confirmation (unless --yes).
     d. Execute `finding.fix_action(snapshot.provider, snapshot.registrar)`.
     e. Report results.
  8. For unfixable but actionable, print detailed instructions.
```

### 4.4 Fix Actions

Fix actions are standalone async functions that receive the provider(s) and affected records. Examples:

- **Remove duplicate records:** Choose the “best” canonical version (e.g., the one with proper quotation), delete the others via `provider.delete_record(zone, record_id)`. If provider does not expose record IDs, use name+type+content matching.
- **Delete stale NS records:** `provider.delete_record(zone, apex_ns_record)` for all non‑delegated NS. Re‑fetch to confirm.
- **Update delegation:** `registrar.assign_nameservers(domain, provider_default_ns)`.

Fix actions must be **idempotent** and log every API call.

### 4.5 Normalizer Module

Critical for duplicate detection. Function `canonical_content(record_type: str, content: str, origin: str) -> str` normalises content following RFC rules for that type. Includes:

- TXT: strip outer `"`, unescape `\"` → `"`, strip whitespace.
- Names (CNAME, NS, MX, PTR, SRV, etc.): make absolute using origin, lower‑case, ensure trailing dot.
- A/AAAA: trim, normalise.
- All: return string for equality comparison.

### 4.6 Safety & Idempotency

- `--fix` will **never** add new records that could alter production behaviour without explicit opt‑in (e.g., email security records). Fixes are restricted to deletions (duplicates, stale NS) and delegation corrections (which restore consistency). For adding records, the tool prints a clear template and instructions; future versions may introduce a `--prescribe` mode for assisted creation.
- Dry‑run mode (`--fix --dry-run`) shows exact API calls that would be made.
- Operations are performed sequentially with short delays to respect rate limits.
- The command refuses to run if provider credentials are invalid/missing (fast‑fail).

## 5. Detailed Check Specifications

### 5.1 Duplicate Records (`duplicates.py`)

**Run:** All records grouped by (normalised name, type, canonical content).  
**Logic:**
1. Build dict `(name_lower, type, canonical) → list[record]`.
2. Any list with length > 1 → `Finding(severity=WARNING, fixable=True)`.
3. Detail: show the raw records and explain why they conflict.
4. Fix: retain one (the first, or prefer one without extra escaping) and delete the rest. If proxied records are involved, preserve the one that matches the user’s intent (heuristic: shortest unescaped content). Provide confirmation output.

**Normalisation example:**
- TXT `"v=spf1 include:_spf.example.com ~all"` and `"\"v=spf1 include:_spf.example.com ~all\""` both normalise to `v=spf1 include:_spf.example.com ~all`. After detection, the fix will delete the escaped version.

### 5.2 Stale NS Records (`ns_stale.py`)

**Run:**
1. Fetch zone apex NS records.
2. Determine reference NS set:
   - If registrar available, use `registrar.read_nameservers(domain)`.
   - Else if provider has a known default NS pattern (e.g., Cloudflare `*.ns.cloudflare.com`), use that.
3. Compare: for each zone NS, if not in reference → `Finding(severity=ERROR, fixable=True if provider allows NS deletion else fixable=False)`.
4. Also warn if no NS records at apex (provider-managed NS may be hidden from API; in that case skip).
5. Fix: delete rogue NS records. If not possible, output dashboard instructions.

### 5.3 Email Security (`email.py`)

Sub‑checks run sequentially; each returns independent findings.

- **MX check:** If NO MX and no NULL MX → `ERROR`. If NULL MX → skip further checks. If multiple MX with same priority → `INFO`.
- **SPF check:** Count TXT apex starting with `v=spf1`.  
  - 0 → `WARNING` (template provided).  
  - >1 → `ERROR` (manual fix required).  
  - Syntax light: check for whitespace between mechanisms, no unknown prefixes. `WARNING` on suspicious syntax.
- **DKIM check:** If MX exists (domain handles email), `WARNING` if no `*._domainkey` TXT with `v=DKIM1`. `INFO` if key < 1024 bits.
- **DMARC check:** `_dmarc` TXT with `v=DMARC1`. Missing → `WARNING`. Multiple → `ERROR`. Parse `p=`; `p=none` → `INFO`; recommend moving to quarantine/reject. Invalid `rua` URI → `WARNING`.
- **MTA‑STS:** `INFO` if missing but MX present.
- **BIMI:** `INFO` if DMARC at quarantine/reject and missing.

**Fix:** None for email checks in initial version. All findings have `fixable=False` with `fix_hint` containing detailed template (e.g., for SPF: `v=spf1 mx -all`). Future `--prescribe` could interactively apply templates.

### 5.4 CNAME at Apex (`cname_apex.py`)

- If apex name has CNAME record → `ERROR`, `fixable=False`. Hint: “Remove the CNAME record; use A/AAAA records or an ALIAS/ANAME solution.”

### 5.5 TTL Check (`ttl.py`)

- For each record (excluding SOA, NS if provider-managed):
  - TTL < 60 → `WARNING` (unless it looks like an ACME challenge: `_acme-challenge` name).
  - TTL > 86400 → `WARNING`.
- `fixable=True` (optionally reset to 3600). In initial version, only report; fix may be added later with user consent.

### 5.6 Delegation Mismatch (`delegation.py`)

- Requires registrar.
- Fetch delegated NS set, compare with zone’s authoritative NS (from provider). If they differ → `ERROR`.
- If registrar credentials not available → `INFO` that delegation check skipped.
- Fix: if both providers operational, `registrar.assign_nameservers(domain, correct_ns)`. Ask for confirmation.

### 5.7 Wildcard & Glue (extended, initially `WARNING` only)

- Glue: for in‑bailiwick NS names, ensure A/AAAA records exist in zone. If missing → `WARNING`, `fixable=False` with instructions.
- Wildcard CNAME → `WARNING`.

## 8. Testing Strategy

1. **Unit tests for `canonical_content`**: verify normalisation of TXT, CNAME, NS, A, AAAA edge cases.
2. **Mocked provider tests** for each check:
   - Inject a `ZoneSnapshot` with controlled records.
   - Verify findings generated, severity, fixable flag.
3. **Integration tests** with real API credentials (to be run manually or in CI with test accounts) to ensure the doctor doesn’t break real zones.
4. **Dry‑run tests** using `--dry-run` to confirm no modifications.
5. **Error handling** when provider/registrar credentials incomplete.

## 10. Acceptance Criteria

- `doctor` runs successfully against a real Cloudflare, GoDaddy, Ionos zone.
- Detects the two stated problems (stale Ionos NS records, duplicated TXT due to quoting) and marks them fixable.
- `--fix --dry-run` shows the deletions without applying.
- `--fix` (without dry‑run) removes the duplicate TXT and stale NS records (if provider allows), or prints explicit manual instructions.
- Email security checks produce correct severity and instructive templates.
- JSON output is valid and machine‑parsable.
- The command refuses destructive actions if credentials are missing and does not create new records without user consent (within `--fix` scope).
